When a new host server registers with the name server the protocol it sends is: HOST:&ip&port

When the host server notifies the name server of a new Agent the protocol is: NEWAGENT:&ID&IP&port&agentComPort

When the name server replies to a new agent is replies NEWAGENT:&name&groupId&GROUPMEM:&ip&port&GROUPMEM:&ip&port //ect

When an agent wants to submit something to the database to be stored for its group the protocol is: STORE&GROUPID&DATA

When an agent wants to contact the database for all its group's data the protocol is: GET&GROUPID

Lunch problem:

I solved the lunch problem as it presented itself in the DIA by making it so that agents could only send a full packet of data to get or store things for their group on the database. Also all the collections that held data I wrapped them with Collections.synchronized() and instantiated them as volatile so that the JVM could do a lot of the thread management. This method of locking out other processes would create a lunch problem if the agents needed to a lot of back and forth with the database, but as I implemented it the single packet being sent creates no back and forth that would hold the serverâ€™s database hostage if someone submitting data decided to go to lunch in the middle of it.   Also in my simple example I allowed the repetition of data.

Sharing data:

I'm sharing data between groups with the DIAdb.java file. This file will run as a server. Agents are the clients. They connect and give their group number. Each group will have their own data structure filled with all the relevant data. This will allow the data to be stored at a central location reachable by each member of the group for display.

Migrating agents:

I migrated the agents basically the same way as in the host server application. The one difference as that the client can't tell these to move they just move randomly 20% of the time someone interacts with them. In a real life application they would only leave if there was a reason to, like if the system sent it a message that it was shutting down soon.

I implemented the agents with a dead link migration approach.  I believe this method to truly be better for this situation than the re-direct migration or zombie migration methods.  In my program the central hub of all interaction is the name server.  This is where the users will find all the current location to the agents.  following the agents around or referring to them by their locations is a bad idea, because they will be migrating.
